결제처럼 공통 필드와 타입별 필드가 함께 있는 도메인에서는
대개 두 전략을 비교하게 된다.

- `Single Table`: 조회는 단순하지만 타입별 `NULL` 컬럼이 늘어난다
- `Joined Table Inheritance`: 스키마는 깔끔하지만 `JOIN` 비용이 생긴다

핵심만 먼저 말하면 이렇다.

> [!TIP]
> 기본 선택지는 보통 `Single Table`이다.
> 타입별 컬럼 차이가 크고, 스키마를 분리해서 관리할 이유가 분명할 때 `Joined`를 고른다.

## 1. 한눈에 비교

| 질문 | `Single Table` | `Joined` |
| --- | --- | --- |
| 부모 기준 목록 조회가 많나 | 유리 | 보통 |
| 타입별 컬럼 차이가 큰가 | 불리 | 유리 |
| `NULL` 컬럼을 줄이고 싶은가 | 불리 | 유리 |
| 타입별 제약을 분리하고 싶은가 | 보통 | 유리 |
| 조회 시 `JOIN`을 줄이고 싶은가 | 유리 | 불리 |

예제 도메인은 아래처럼 잡겠다.

- 모든 결제는 `id`, `order_id`, `amount`, `created_at`를 가진다
- 카드 결제만 `card_last4`, `approval_code`가 필요하다
- 계좌이체만 `bank_name`, `depositor_name`이 필요하다

## 2. Single Table 설계

모든 타입을 한 테이블에 저장하는 방식이다.

```sql
create table payment (
  id bigint primary key generated always as identity, -- 결제 PK
  type varchar(30) not null, -- 결제 타입(card, bank_transfer)
  order_id varchar(50) not null, -- 주문 ID
  amount integer not null, -- 결제 금액
  created_at timestamptz not null default now(), -- 생성 시각
  card_last4 varchar(4), -- 카드 끝 4자리
  approval_code varchar(50), -- 카드 승인 코드
  bank_name varchar(50), -- 이체 은행명
  depositor_name varchar(100) -- 입금자명
);
```

예시 데이터는 바로 이해된다.

| id | type | amount | card_last4 | approval_code | bank_name | depositor_name |
| --- | --- | --- | --- | --- | --- | --- |
| 101 | `card` | 15000 | `4242` | `APPROVED-9A3` | `NULL` | `NULL` |
| 102 | `bank_transfer` | 23000 | `NULL` | `NULL` | `KakaoBank` | `Kim` |

장점

- 목록 조회가 단순하다: 부모와 자식을 구분하지 않고 `payment` 하나만 읽으면 되기 때문이다.
- 다형 조회도 보통 한 테이블에서 끝난다: 타입이 달라도 같은 row 집합 안에서 필드를 바로 꺼낼 수 있다.
- ORM 쿼리가 읽기 쉽다: 조인 전략을 따로 고민하지 않아도 되는 경우가 많다.

주의할 점

- 타입이 늘수록 컬럼과 `NULL`이 많아진다: 카드용, 이체용, 향후 다른 타입용 컬럼이 한 테이블에 계속 추가되기 때문이다.
- 타입별 `NOT NULL` 제약을 강하게 걸기 어렵다: 카드 결제에서는 필수인 컬럼이 이체 결제에서는 비어 있어야 하므로 공통 스키마에서 일관되게 강제하기가 애매하다.
- 테이블 의미가 점점 흐려질 수 있다: `payment`가 공통 결제 테이블인지, 모든 타입별 상세를 다 담는 거대한 테이블인지 경계가 흐려진다.

## 3. Joined Table Inheritance 설계

공통 컬럼은 부모 테이블에, 타입별 컬럼은 자식 테이블에 둔다.

```sql
create table payment (
  id bigint primary key generated always as identity, -- 결제 PK
  type varchar(30) not null, -- 결제 타입(card, bank_transfer)
  order_id varchar(50) not null, -- 주문 ID
  amount integer not null, -- 결제 금액
  created_at timestamptz not null default now() -- 생성 시각
);

create table card_payment (
  id bigint primary key references payment(id), -- payment.id를 공유하는 PK
  card_last4 varchar(4) not null, -- 카드 끝 4자리
  approval_code varchar(50) not null -- 카드 승인 코드
);

create table bank_transfer_payment (
  id bigint primary key references payment(id), -- payment.id를 공유하는 PK
  bank_name varchar(50) not null, -- 이체 은행명
  depositor_name varchar(100) not null -- 입금자명
);
```

예시 데이터는 이렇게 나뉜다.

`payment`

| id | type | amount |
| --- | --- | --- |
| 101 | `card` | 15000 |
| 102 | `bank_transfer` | 23000 |

`card_payment`

| id | card_last4 | approval_code |
| --- | --- | --- |
| 101 | `4242` | `APPROVED-9A3` |

`bank_transfer_payment`

| id | bank_name | depositor_name |
| --- | --- | --- |
| 102 | `KakaoBank` | `Kim` |

장점

- 타입별 컬럼과 제약을 분리할 수 있다: 카드 관련 필수값은 `card_payment`에, 이체 관련 필수값은 `bank_transfer_payment`에 직접 걸 수 있다.
- `NULL` 컬럼이 줄어든다: 어떤 타입에 필요 없는 컬럼을 부모 테이블에 억지로 두지 않기 때문이다.
- 스키마만 봐도 타입 구조가 드러난다: 부모는 공통 상태, 자식은 타입별 상세라는 역할이 테이블 이름과 컬럼 배치에 바로 보인다.

주의할 점

- 자식 상세를 읽을 때 `JOIN`이 필요하다: 카드 승인 코드 같은 정보는 부모 테이블에 없고 자식 테이블에만 있기 때문이다.
- 자식 타입이 늘수록 다형 조회가 무거워질 수 있다: 부모 기준으로 전체 상세를 읽을 때 자식 테이블들을 함께 붙여야 하기 때문이다.
- 쓰기 시에도 부모와 자식 insert가 나뉜다: 하나의 결제를 저장해도 실제로는 부모 row와 자식 row를 모두 기록해야 한다.

## 4. SQLAlchemy 매핑 차이

### 4.1 Single Table

```python
class Payment(Base):
    __tablename__ = "payment"

    id: Mapped[int] = mapped_column(primary_key=True)  # 결제 PK
    type: Mapped[str] = mapped_column(String(30))  # 결제 타입
    order_id: Mapped[str] = mapped_column(String(50), index=True)  # 주문 ID
    amount: Mapped[int]  # 결제 금액
    card_last4: Mapped[str | None]  # 카드 끝 4자리
    approval_code: Mapped[str | None]  # 카드 승인 코드
    bank_name: Mapped[str | None]  # 이체 은행명
    depositor_name: Mapped[str | None]  # 입금자명

    __mapper_args__ = {"polymorphic_on": type}


class CardPayment(Payment):
    __mapper_args__ = {"polymorphic_identity": "card"}


class BankTransferPayment(Payment):
    __mapper_args__ = {"polymorphic_identity": "bank_transfer"}
```

핵심은 간단하다.

- 테이블은 하나다
- 자식 클래스는 같은 테이블을 공유한다
- 조회는 단순하지만 자식 컬럼이 부모 테이블에 같이 들어간다

코드를 읽을 때는 아래 네 줄이 중요하다.

- `__tablename__ = "payment"`: 부모와 자식이 결국 같은 `payment` 테이블을 공유한다.
- `polymorphic_on = type`: `payment.type` 값을 보고 어떤 자식 클래스로 만들지 결정한다.
- `CardPayment.polymorphic_identity = "card"`: `type='card'`인 row는 `CardPayment`로 해석한다.
- 자식에 `__tablename__`이 없다: 별도 자식 테이블이 없다는 뜻이다.

### 4.2 Joined

```python
class Payment(Base):
    __tablename__ = "payment"

    id: Mapped[int] = mapped_column(primary_key=True)  # 결제 PK
    type: Mapped[str] = mapped_column(String(30))  # 결제 타입
    order_id: Mapped[str] = mapped_column(String(50), index=True)  # 주문 ID
    amount: Mapped[int]  # 결제 금액

    __mapper_args__ = {
        "polymorphic_on": type,
        "polymorphic_identity": "payment",
    }


class CardPayment(Payment):
    __tablename__ = "card_payment"

    id: Mapped[int] = mapped_column(ForeignKey("payment.id"), primary_key=True)  # 부모 PK 공유
    card_last4: Mapped[str] = mapped_column(String(4))  # 카드 끝 4자리
    approval_code: Mapped[str] = mapped_column(String(50))  # 카드 승인 코드

    __mapper_args__ = {"polymorphic_identity": "card"}


class BankTransferPayment(Payment):
    __tablename__ = "bank_transfer_payment"

    id: Mapped[int] = mapped_column(ForeignKey("payment.id"), primary_key=True)  # 부모 PK 공유
    bank_name: Mapped[str] = mapped_column(String(50))  # 이체 은행명
    depositor_name: Mapped[str] = mapped_column(String(100))  # 입금자명

    __mapper_args__ = {"polymorphic_identity": "bank_transfer"}
```

핵심 차이는 딱 두 줄이다.

- 자식에 `__tablename__`이 있다
- 자식 `id`가 부모 `payment.id`를 참조한다

코드를 읽을 때는 아래 포인트를 같이 보면 된다.

- 부모 `payment`는 공통 필드만 가진다.
- 자식 `card_payment`, `bank_transfer_payment`는 타입별 상세 필드만 가진다.
- 자식 `id`가 부모 PK를 그대로 공유하므로, 의미상 하나의 결제를 여러 테이블에 나눠 저장하는 구조가 된다.
- 그래서 `CardPayment` 객체를 만들려면 부모 row와 자식 row를 합쳐 읽어야 한다.

### 4.3 왜 조인 문법이 없어도 JOIN이 생기나

`select(CardPayment)`에 조인 문법이 없어 보여도,
ORM은 `CardPayment`가 어떤 테이블 구조에 매핑돼 있는지 이미 알고 있다.

| 전략 | ORM이 아는 구조 | 그래서 생기는 SQL 경로 |
| --- | --- | --- |
| `Single Table` | `CardPayment`의 필드가 전부 `payment`에 있다 | `payment`만 읽는다 |
| `Joined` | 공통 필드는 `payment`, 카드 필드는 `card_payment`에 있다 | `payment`와 `card_payment`를 함께 읽는다 |

즉, SQLAlchemy는 "조인 문법이 없는데도 JOIN을 해주는 것"이 아니라
"해당 엔티티를 만들기 위해 필요한 테이블을 매핑 정보로부터 계산하는 것"에 가깝다.

### 5.1 카드 결제만 조회

`Single Table`

```python
stmt = (
    select(CardPayment)
    .where(CardPayment.amount >= 10000)
    .order_by(CardPayment.id.desc())
)
rows = list((await session.scalars(stmt)).all())
```

`Joined`

```python
stmt = (
    select(CardPayment)
    .where(CardPayment.amount >= 10000)
    .order_by(CardPayment.id.desc())
)
rows = list((await session.scalars(stmt)).all())
```

같은 Python 코드처럼 보여도 읽는 경로는 다르다.

| 전략 | 실제 읽는 테이블 |
| --- | --- |
| `Single Table` | `payment` 하나 |
| `Joined` | `payment + card_payment` |

대략 이런 차이다.

`Single Table`

```sql
select *
from payment
where type = 'card'
  and amount >= 10000;
```

`Joined`

```sql
select ...
from payment
join card_payment on card_payment.id = payment.id
where payment.amount >= 10000;
```

### 5.2 부모 공통 컬럼만 목록 조회

실무에서는 목록 화면에서 자식 상세가 꼭 필요하지 않은 경우가 많다.

`Single Table`

```python
stmt = (
    select(Payment.id, Payment.type, Payment.order_id, Payment.amount)
    .order_by(Payment.id.desc())
)
rows = list((await session.execute(stmt)).all())
```

`Joined`

```python
stmt = (
    select(Payment.id, Payment.type, Payment.order_id, Payment.amount)
    .order_by(Payment.id.desc())
)
rows = list((await session.execute(stmt)).all())
```

이 경우는 오히려 차이가 거의 없다.

| 전략 | 실제 읽는 테이블 |
| --- | --- |
| `Single Table` | `payment` 하나 |
| `Joined` | `payment` 하나 |

즉 `Joined`를 쓴다고 해서 모든 조회에 JOIN이 붙는 것은 아니다.
부모 공통 컬럼만 읽으면 부모 테이블만 읽게 만들 수 있다.

### 5.3 자식 전용 필드로 필터링

카드 승인 코드처럼 자식 전용 컬럼으로 조건을 거는 상황이다.

`Single Table`

```python
stmt = (
    select(CardPayment)
    .where(CardPayment.approval_code.like("APPROVED%"))
    .order_by(CardPayment.id.desc())
)
rows = list((await session.scalars(stmt)).all())
```

`Joined`

```python
stmt = (
    select(CardPayment)
    .where(CardPayment.approval_code.like("APPROVED%"))
    .order_by(CardPayment.id.desc())
)
rows = list((await session.scalars(stmt)).all())
```

여기서도 Python 코드는 같지만 차이는 유지된다.

| 전략 | 실제 읽는 테이블 |
| --- | --- |
| `Single Table` | `payment` 하나 |
| `Joined` | `payment + card_payment` |

이유는 단순하다.

- `Single Table`에서는 `approval_code`가 `payment`에 있다
- `Joined`에서는 `approval_code`가 `card_payment`에 있다

### 5.4 모든 결제를 상세까지 조회

`Single Table`

```python
stmt = select(Payment).order_by(Payment.id.desc())
rows = list((await session.scalars(stmt)).all())
```

`Joined`

```python
payment_entity = with_polymorphic(
    Payment,
    [CardPayment, BankTransferPayment],
)

stmt = select(payment_entity).order_by(payment_entity.id.desc())
rows = list((await session.scalars(stmt)).all())
```

차이는 더 직접적이다.

| 전략 | 실제 읽는 테이블 |
| --- | --- |
| `Single Table` | `payment` 하나 |
| `Joined` | `payment + 모든 자식 테이블` |

이 상황이 `Joined`의 비용이 가장 잘 드러나는 구간이다.
자식 타입이 늘어나면 전체 상세 조회도 그만큼 무거워질 수 있다.

### 5.5 특정 결제 한 건 상세 조회

운영 화면에서 `payment_id` 하나를 눌러 상세를 보는 상황이다.

`Single Table`

```python
stmt = select(Payment).where(Payment.id == payment_id)
payment = await session.scalar(stmt)
```

`Joined`

```python
payment_entity = with_polymorphic(
    Payment,
    [CardPayment, BankTransferPayment],
)

stmt = select(payment_entity).where(payment_entity.id == payment_id)
payment = await session.scalar(stmt)
```

| 전략 | 실제 읽는 테이블 |
| --- | --- |
| `Single Table` | `payment` 하나 |
| `Joined` | `payment + 필요한 자식 테이블들` |

목록이 아니라 상세 API라면 이 비용은 받아들일 만한 경우가 많다.
그래서 `Joined`를 쓸 때는 목록과 상세를 분리하는 설계가 중요하다.

### 5.6 카드 결제 생성

`Single Table`

```python
payment = CardPayment(
    order_id="ORD-2026-0001",
    amount=15000,
    card_last4="4242",
    approval_code="APPROVED-9A3",
)
session.add(payment)
await session.commit()
```

`Joined`

```python
payment = CardPayment(
    order_id="ORD-2026-0001",
    amount=15000,
    card_last4="4242",
    approval_code="APPROVED-9A3",
)
session.add(payment)
await session.commit()
```

여기도 Python 코드는 비슷하지만 DB 쓰기 경로가 다르다.

| 전략 | 실제 쓰기 경로 |
| --- | --- |
| `Single Table` | `payment`에 insert 1회 |
| `Joined` | `payment` insert 후 `card_payment` insert |

즉 쓰기까지 포함해서 보면 차이는 이렇게 요약된다.

| 상황 | `Single Table` | `Joined` |
| --- | --- | --- |
| 자식 하나 조회 | 단일 테이블 조회 | 부모 + 자식 JOIN |
| 부모 목록 조회 | 단일 테이블 조회 | 부모만 읽으면 JOIN 없이 가능 |
| 전체 상세 조회 | 단일 테이블 조회 | 자식 수만큼 JOIN 가능성 |
| 생성 | insert 1회 | 부모 insert + 자식 insert |

## 6. Joined의 트레이드오프

Joined는 `깔끔한 스키마`를 얻는 대신 `조회와 쓰기 경로가 길어지는` 전략이다.
실제로 비용이 어디서 생기는지는 아래처럼 정리할 수 있다.

| 항목 | 얻는 것 | 치르는 비용 |
| --- | --- | --- |
| 스키마 구조 | 공통 필드와 타입별 필드가 분리된다 | 테이블 수가 늘고 구조를 한번 더 따라가야 한다 |
| 데이터 무결성 | 타입별 필수값을 자식 테이블에 명확히 둘 수 있다 | 다형 모델을 이해하지 못하면 쿼리 작성이 오히려 어려워진다 |
| 목록 조회 | 부모 컬럼만 읽을 땐 비교적 깔끔하다 | 상세까지 같이 읽는 순간 JOIN 비용이 생긴다 |
| 상세 조회 | 타입별 정보가 명확하게 모인다 | 자식 타입 수가 늘수록 전체 상세 조회가 무거워진다 |
| 쓰기 | 부모/자식 역할이 분리돼 의미가 선명하다 | insert, update, delete가 여러 테이블에 걸칠 수 있다 |
| 운영 | 스키마만 봐도 도메인 구조가 보인다 | 디버깅할 때 실제 SQL과 실행 계획을 더 자주 봐야 한다 |

특히 실무에서는 아래 세 가지를 자주 체감한다.

- 부모 목록은 괜찮은데 상세 화면이나 관리자 조회가 느려질 수 있다: 대부분 여기서 `with_polymorphic`나 다형 JOIN 비용이 드러난다.
- 새 타입이 추가될수록 코드보다 조회 전략이 더 중요해진다: 스키마 분리는 쉬워도 "어떤 API가 어디까지 상세를 읽어야 하는가"를 다시 정리해야 한다.
- ORM이 추상화를 주지만 비용을 없애주지는 않는다: Python 코드가 단순해 보여도 실제 SQL은 부모/자식 조인으로 바뀔 수 있다.

그래서 Joined는 "정규화가 예쁘다"만으로 고르기보다는,
아래 질문에 `예`라고 답할 때 고르는 편이 안전하다.

- 타입별 필드 차이가 충분히 큰가
- 타입별 제약을 스키마에 분리할 가치가 큰가
- 상세 조회 비용보다 모델 명확성이 더 중요한가

## 7. 그래서 언제 Joined를 쓰나

`Joined`가 맞는 경우

- 타입별 컬럼 차이가 크다
- 타입별 제약을 스키마에 분리하고 싶다
- 부모와 자식을 같은 정체성으로 다뤄야 한다
- 조회 단순성보다 스키마 명확성이 더 중요하다

`Single Table`이 더 나은 경우

- 부모 기준 목록 조회가 많다
- 빠른 CRUD와 단순 쿼리가 우선이다
- 자식 타입이 자주 늘어난다
- `NULL` 컬럼을 감수해도 읽기 단순성이 더 중요하다

> [!NOTE]
> 객체에 상속이 보인다고 해서 DB도 상속 매핑으로 가야 하는 것은 아니다.
> 기본값은 보통 더 단순한 쪽이고,
> `Joined`는 필요가 분명할 때 선택하는 전략에 가깝다.

## 8. 결론

두 전략의 차이는 명확하다.

- `Single Table`: 읽기 단순성에 강하다
- `Joined`: 스키마 분리와 타입별 제약에 강하다

즉, `Joined Table Inheritance`는
상속을 예쁘게 표현하는 기능이라기보다
**조회 단순성을 일부 포기하고 스키마 명확성을 얻는 선택**이다.
