객체지향을 배웠는데 코드가 더 복잡해졌다는 말은 흔하다.
`class`, `extends`, `interface`, `SOLID`를 붙였는데도 수정은 더 어려워지기 때문이다.
문제는 대개 객체지향을 문법으로 먼저 배우고, 변경을 줄이는 구조로는 나중에 배우는 데 있다.

실무에서 중요한 질문은 "클래스를 몇 개 만들었는가"가 아니다.
새 결제 수단이 추가될 때 어디를 고쳐야 하는가, 주문 상태 규칙이 바뀔 때 어떤 코드가 함께 흔들리는가가 더 중요하다.
이 글에서는 주문/결제 예제를 기준으로 객체지향을 다시 정리한다.

핵심은 아래 세 문장이다.

- `객체지향은 클래스 문법이 아니라 변경을 다루는 방식이다.`
- `좋은 객체는 상태보다 책임이 먼저 보인다.`
- `유연한 설계는 상속보다 추상화와 다형성에서 더 자주 나온다.`

## 이 글에서 정리할 질문

- 왜 절차 중심 코드는 요구사항이 늘수록 수정 범위가 커지는가
- 객체는 데이터를 묶는 그릇이 아니라 무엇을 책임져야 하는가
- 상속과 포함 관계는 어떤 기준으로 나눠야 하는가
- 다형성은 왜 분기문을 줄이고 확장을 쉽게 만드는가
- SOLID는 실무에서 어떤 판단 기준으로 읽어야 하는가

## 1. 왜 절차를 나누는 것만으로는 부족해지는가

프로그래밍은 처음부터 객체지향이었던 것이 아니다.
초기에는 하드웨어를 어떻게 제어할지, 그 다음에는 큰 프로그램을 어떻게 절차로 나눌지, 그 다음에는 커지는 변경을 어떻게 통제할지가 더 중요해졌다.

정리하면 패러다임의 초점은 대략 이렇게 이동했다.

- 기계 중심: CPU와 메모리가 이해하는 명령을 어떻게 정확히 써야 하는가
- 절차 중심: 큰 작업을 함수와 순서로 어떻게 쪼갤 것인가
- 구조 중심: `goto`를 줄이고 흐름을 더 읽기 쉽게 만들 수 있는가
- 객체 중심: 데이터와 그 데이터를 다루는 규칙을 어떻게 함께 움직이게 만들 것인가

절차를 나누는 방식은 여전히 중요하다.
문제는 서비스가 커질수록 "순서"보다 "변경 이유"가 더 큰 복잡성을 만든다는 점이다.

예를 들어 주문 결제 기능이 처음에는 단순할 수 있다.

- 카드 결제만 지원한다
- 주문 상태는 `READY`, `PAID` 두 개뿐이다
- 승인 실패 시 바로 예외를 던진다

하지만 금방 요구사항이 늘어난다.

- 계좌이체와 간편결제가 추가된다
- 결제 전 재고 확인 규칙이 붙는다
- 주문 상태가 `CANCELED`, `REFUNDED`, `PARTIALLY_PAID`처럼 늘어난다
- 결제 수단마다 승인 규칙과 로깅 규칙이 달라진다

이때 함수만 잘게 쪼개 두었다고 해서 변경이 자동으로 쉬워지지는 않는다.
같은 데이터가 여러 함수로 흩어지고, 한 규칙을 바꾸기 위해 여러 곳을 함께 수정해야 하기 때문이다.

결국 객체지향이 다루려는 문제는 "함수를 더 예쁘게 나누는 방법"이 아니라 "함께 변해야 하는 것들을 어디에 묶을 것인가"에 가깝다.

## 2. 데이터와 로직을 따로 두면 어디서 복잡해지는가

절차지향과 객체지향의 차이를 문법으로만 보면 핵심이 빠진다.
실제로는 데이터와 규칙이 서로 얼마나 가까이 붙어 있는지가 더 중요하다.

주문 결제를 절차적으로 먼저 써 보자.

```python
def pay_order(order: dict, payment_type: str, payload: dict) -> None:
    if order['status'] != 'READY':
        raise ValueError('already processed')

    if payment_type == 'card':
        validate_card(payload)
        approved = request_card_gateway(payload, order['total_amount'])
    elif payment_type == 'bank_transfer':
        validate_bank_account(payload)
        approved = request_bank_gateway(payload, order['total_amount'])
    else:
        raise ValueError('unsupported payment type')

    if not approved:
        raise ValueError('payment failed')

    order['status'] = 'PAID'
    order['paid_amount'] = order['total_amount']
```

작은 프로그램이라면 이 코드도 충분히 이해 가능하다.
문제는 요구사항이 늘어날 때다.
새 결제 수단이 들어오면 조건문을 수정해야 하고, 주문 상태 규칙이 바뀌면 딕셔너리 필드와 여러 헬퍼 함수를 함께 손봐야 한다.

아래 표로 보면 차이가 더 분명하다.

| 관점                   | 절차 중심 설계                     | 객체 중심 설계                      |
| ---------------------- | ---------------------------------- | ----------------------------------- |
| 변경 기준              | 함수 흐름과 분기                   | 책임을 가진 객체                    |
| 데이터 위치            | 여러 함수 인자와 딕셔너리에 흩어짐 | 객체 내부 상태로 모임               |
| 규칙 변경 시 수정 범위 | 여러 함수와 분기를 함께 수정       | 해당 책임을 가진 객체 중심으로 수정 |
| 새 결제 수단 추가      | `if/elif`와 검증 로직 추가         | 새 구현체 추가 후 협력에 연결       |
| 테스트 초점            | 큰 함수의 여러 분기                | 객체 단위의 책임 검증               |

절차형 코드가 나쁘다는 뜻은 아니다.
규칙이 단순하고 수명이 짧은 스크립트라면 절차형이 오히려 명확하다.
다만 주문, 결제, 환불처럼 상태와 규칙이 함께 커지는 도메인에서는 데이터와 로직을 너무 멀리 떼어 놓는 순간 복잡성이 빨리 올라간다.

객체지향은 여기서 출발한다.
같이 변하는 상태와 규칙을 한곳에 모아 수정 범위를 줄이려는 시도다.

## 3. 설계도보다 중요한 것은 인스턴스의 책임이다

객체지향 입문에서 클래스는 흔히 "설계도", 객체는 "실체"라고 배운다.
완전히 틀린 설명은 아니지만, 그 정의만으로는 왜 객체가 필요한지까지는 잘 보이지 않는다.

실무적으로 더 중요한 질문은 이것이다.

- 이 객체는 어떤 상태를 관리하는가
- 어떤 메시지에 응답해야 하는가
- 어떤 규칙을 자기 내부에서 책임져야 하는가

아래 코드는 주문과 결제 수단의 책임을 나눈 예시다.

```python
from abc import ABC, abstractmethod
from dataclasses import dataclass


class PaymentMethod(ABC):
    @abstractmethod
    def pay(self, amount: int) -> None:
        ...


@dataclass
class Order:
    id: str
    total_amount: int
    status: str = 'READY'

    def pay(self, payment_method: PaymentMethod) -> None:
        if self.status != 'READY':
            raise ValueError('already processed')

        payment_method.pay(self.total_amount)
        self.status = 'PAID'
```

여기서 중요한 점은 `Order`가 자기 상태를 스스로 바꾼다는 것이다.
외부 함수가 `order['status'] = 'PAID'`를 직접 만지는 대신, `Order.pay()`가 상태 변경 규칙을 가진다.

즉 객체는 데이터를 담는 상자가 아니라 "상태와 규칙이 함께 있는 책임 단위"로 보는 편이 정확하다.
그래서 객체지향의 핵심은 클래스 개수가 아니라 객체 간 협력 구조에 있다.

- `Order`는 주문 상태 전이를 책임진다
- `PaymentMethod`는 결제 승인 방법을 책임진다
- 둘은 메시지 호출로 협력한다

좋은 객체는 상태보다 책임이 먼저 보인다.
상태 필드 목록만 길고 행동이 거의 없다면, 그 객체는 아직 도메인 규칙을 충분히 끌어안지 못했을 가능성이 크다.

## 4. 상속으로 묶을 것인가, 포함으로 연결할 것인가

객체지향에서 자주 헷갈리는 부분이 `is-a`와 `has-a`다.
겉으로 비슷해 보여도 설계 선택은 전혀 다르다.

- `is-a`: 어떤 타입이 다른 타입의 한 종류인가
- `has-a`: 어떤 객체가 다른 객체를 속성이나 협력 대상으로 가지는가

주문/결제 예제로 보면 아래처럼 읽는 것이 자연스럽다.

- `CardPayment is-a PaymentMethod`
- `BankTransferPayment is-a PaymentMethod`
- `Order has-a PaymentMethod`

```python
class PaymentMethod(ABC):
    @abstractmethod
    def pay(self, amount: int) -> None:
        ...


class CardPayment(PaymentMethod):
    def pay(self, amount: int) -> None:
        print(f'card approved: {amount}')


class BankTransferPayment(PaymentMethod):
    def pay(self, amount: int) -> None:
        print(f'bank transfer requested: {amount}')


@dataclass
class Order:
    id: str
    payment_method: PaymentMethod
```

여기서 `CardPayment`와 `BankTransferPayment`는 `PaymentMethod`의 한 종류이므로 상속이 자연스럽다.
반면 `Order`는 `PaymentMethod`의 한 종류가 아니라, 결제를 위해 그 수단을 가진다.
그래서 `Order`와 `PaymentMethod`의 관계는 포함 또는 합성에 가깝다.

> [!NOTE]
> `is-a`는 단순히 "비슷해 보인다"가 아니라 "부모 타입 자리에 넣어도 동작 의미가 유지되는가"로 판단해야 한다.
> 이 기준이 없으면 상속은 재사용 도구가 아니라 결합도만 높이는 장치가 된다.

상속은 강한 연결이다.
부모의 규약, 생성 방식, 상태 구조에 자식이 묶이기 쉽다.
그래서 실무에서는 보통 먼저 `has-a`를 검토하고, 진짜 치환 가능성이 분명할 때만 `is-a`를 선택하는 편이 안전하다.

유연한 설계는 상속보다 추상화와 다형성에서 더 자주 나온다.

## 5. 객체지향을 이루는 네 가지 축은 무엇인가

객체지향의 4대 요소는 흔히 추상화, 캡슐화, 상속, 다형성으로 정리한다.
문제는 이 네 가지를 정의만 외우고 실제 설계와 연결하지 못하는 경우가 많다는 점이다.

이 글에서는 아래처럼 이해하면 충분하다.

- 추상화: 지금 중요한 개념만 남기고 나머지 세부사항을 뒤로 미루는 것
- 캡슐화: 상태와 그 상태를 바꾸는 규칙을 한곳에 모으는 것
- 상속: 공통 규약을 공유하는 타입 계층을 만드는 것
- 다형성: 같은 메시지를 다른 구현으로 처리하게 만드는 것

핵심은 네 요소가 서로 따로 노는 개념이 아니라는 점이다.
추상화로 공통 모델을 만들고, 캡슐화로 규칙을 모으고, 필요할 때만 상속으로 타입을 정리하고, 다형성으로 확장을 연다.

이 순서가 중요한 이유는 상속만 먼저 배우면 객체지향을 "부모-자식 트리 만들기"로 오해하기 쉽기 때문이다.
하지만 실무에서 더 자주 쓰는 축은 보통 아래 둘이다.

- 어떤 책임을 어떤 객체로 추상화할 것인가
- 그 책임을 어떻게 캡슐화하고 다형성으로 확장할 것인가

즉 객체지향의 중심은 "계층도"보다 "책임과 협력"에 있다.

## 6. 좋은 객체는 무엇을 숨기고 무엇을 드러내야 하는가

캡슐화를 `private` 필드와 `getter/setter`로만 이해하면 금방 한계에 부딪힌다.
실제로 중요한 것은 "무엇을 숨겼는가"보다 "변경 이유를 어디에 모았는가"다.

아래 주문 객체를 보자.

```python
from dataclasses import dataclass


@dataclass
class Order:
    subtotal: int
    shipping_fee: int
    status: str = 'READY'
    coupon_discount: int = 0

    def apply_coupon(self, coupon) -> None:
        if self.status != 'READY':
            raise ValueError('cannot change paid order')

        self.coupon_discount = coupon.calculate_discount(self.subtotal)

    @property
    def total_amount(self) -> int:
        return self.subtotal + self.shipping_fee - self.coupon_discount
```

이 예시에서 외부는 `subtotal`, `shipping_fee`, `coupon_discount`를 제멋대로 조작할 필요가 없다.
대신 주문 금액 규칙은 `apply_coupon()`과 `total_amount` 계산에 모인다.

좋은 객체는 모든 데이터를 꽁꽁 숨기는 객체가 아니다.
바깥에 공개해야 할 모델은 드러내되, 상태 전이와 계산 규칙은 내부에 모아 둔다.

추상화도 같은 맥락으로 읽을 수 있다.
추상화는 세부 구현을 무조건 감추는 마법이 아니라, 지금 협력에 필요한 관점만 드러내는 일이다.

- 주문은 "결제 가능한 상태인가"를 드러내면 된다
- 결제 수단은 "얼마를 승인할 수 있는가"를 드러내면 된다
- 카드 번호 검증 방식이나 외부 PG API 세부 필드는 그 뒤에 감출 수 있다

즉 좋은 객체는 "내부 데이터 구조"보다 "외부에 제공하는 안정된 약속"이 먼저 보인다.
그 약속이 선명할수록 구현 변경이 바깥으로 덜 새어 나온다.

## 7. 분기문을 줄이고 확장을 여는 방법

다형성은 종종 어려운 개념처럼 보이지만, 실무에서는 꽤 단순한 문제를 해결한다.
새 타입을 추가할 때 기존 코드를 계속 뜯어고쳐야 하는 상황을 줄이는 것이다.

앞에서 본 절차형 결제 함수의 핵심 문제는 `payment_type` 분기다.
결제 수단이 늘어날수록 `if/elif`가 늘고, 검증 로직과 외부 연동 코드도 한 함수 안에 모인다.

다형성을 적용하면 중심 로직은 "무엇을 할 것인가"만 남기고, "어떻게 할 것인가"는 구현체로 밀어낼 수 있다.

```python
from abc import ABC, abstractmethod


class PaymentMethod(ABC):
    @abstractmethod
    def authorize(self, amount: int) -> str:
        ...


class CardPayment(PaymentMethod):
    def authorize(self, amount: int) -> str:
        return f'card-approval:{amount}'


class KakaoPayPayment(PaymentMethod):
    def authorize(self, amount: int) -> str:
        return f'kakaopay-approval:{amount}'


def checkout(order: Order, payment_method: PaymentMethod) -> None:
    approval_id = payment_method.authorize(order.total_amount)
    order.status = 'PAID'
    print(f'payment completed: {approval_id}')
```

이제 새 결제 수단 `NaverPayPayment`를 추가해도 `checkout()`은 바뀌지 않는다.
중심 로직이 결제 수단의 구체 타입을 직접 알지 않기 때문이다.

> [!TIP]
> 새 타입을 추가할 때마다 기존 함수의 `if/elif`를 열어야 한다면, 타입별 책임이 아직 객체 뒤로 충분히 이동하지 않았다는 신호일 가능성이 크다.

물론 분기문이 항상 나쁜 것은 아니다.
타입이 둘뿐이고 앞으로도 늘지 않을 것이 분명하다면 분기문이 가장 단순할 수 있다.
하지만 확장 가능성이 있는 도메인에서 다형성은 분기를 줄이는 동시에 테스트 대체도 쉽게 만든다.

```python
class FakePayment(PaymentMethod):
    def __init__(self):
        self.authorized_amount = None

    def authorize(self, amount: int) -> str:
        self.authorized_amount = amount
        return 'fake-approval'


order = Order(subtotal=10000, shipping_fee=2500)
payment = FakePayment()
checkout(order, payment)

assert payment.authorized_amount == 12500
assert order.status == 'PAID'
```

이 테스트가 쉬운 이유도 같다.
`checkout()`은 구체 결제 수단 대신 `PaymentMethod`라는 추상화에만 의존하기 때문이다.

객체지향은 클래스 문법이 아니라 변경을 다루는 방식이다.
다형성은 그중에서도 "추가될 것을 미리 열어 두는" 가장 실용적인 기술 중 하나다.

## 8. SOLID는 규칙이 아니라 변경 비용을 줄이는 기준이다

SOLID는 종종 체크리스트처럼 소비된다.
하지만 실무에서는 "이 원칙을 암기했는가"보다 "어떤 변경 비용을 줄이려는가"로 읽는 편이 훨씬 유용하다.

| 원칙 | 언제 필요한가                                    | 줄이려는 비용                                 | 흔한 오해                                |
| ---- | ------------------------------------------------ | --------------------------------------------- | ---------------------------------------- |
| SRP  | 하나의 객체가 여러 이유로 자주 같이 수정될 때    | 한 변경이 여러 책임을 동시에 깨뜨리는 비용    | 클래스는 메서드가 하나여야 한다          |
| OCP  | 새 타입이 반복적으로 추가될 때                   | 기존 코드를 자꾸 수정하는 비용                | 기존 코드를 절대 바꾸면 안 된다          |
| LSP  | 상속 구조를 만들었는데 치환이 불안할 때          | 부모 타입 사용 코드가 자식 때문에 깨지는 비용 | 상속만 했으면 자동으로 성립한다          |
| ISP  | 큰 인터페이스 때문에 불필요한 메서드가 따라올 때 | 구현체가 필요 없는 약속까지 떠안는 비용       | 인터페이스는 무조건 잘게 쪼개야 한다     |
| DIP  | 핵심 로직이 외부 기술 세부사항에 묶일 때         | 테스트와 교체가 어려워지는 비용               | 인터페이스를 많이 만들면 무조건 좋아진다 |

각 원칙을 주문/결제 예제로 짧게 연결해 보면 아래처럼 읽을 수 있다.

- `SRP`: 주문 상태 전이와 결제 승인 호출을 한 객체가 다 떠안으면 변경 이유가 섞인다.
- `OCP`: 새 결제 수단을 추가할 때 `checkout()` 대신 새 구현체를 추가하는 쪽이 덜 흔들린다.
- `LSP`: `PaymentMethod`를 상속했는데 어떤 구현체는 금액 승인 대신 예외만 던진다면 치환 가능성이 깨진다.
- `ISP`: 환불을 지원하지 않는 결제 수단까지 `refund()`를 강제하는 인터페이스는 과하다.
- `DIP`: 주문 서비스가 특정 카드사 SDK를 직접 호출하면 테스트와 교체가 어려워진다.

중요한 것은 SOLID가 "객체지향을 멋있게 보이게 만드는 장식"이 아니라는 점이다.
이 원칙들은 결국 아래 질문으로 돌아온다.

- 어디가 자주 변하는가
- 그 변화가 왜 여러 곳으로 퍼지는가
- 어떤 추상화가 그 전파를 줄일 수 있는가

정리하면 SOLID는 정답 세트가 아니다.
변경 비용을 줄이기 위해 설계를 점검하는 질문 세트에 가깝다.

결국 객체지향을 잘 쓴다는 것은 클래스 계층을 복잡하게 만드는 일이 아니다.
주문은 주문답게, 결제 수단은 결제 수단답게 자기 책임을 가지게 두고, 함께 바뀌는 규칙을 가까이 두는 일이다.
그렇게 해야 절차가 길어져서가 아니라 변경 이유가 퍼져서 생기는 복잡성을 줄일 수 있다.
