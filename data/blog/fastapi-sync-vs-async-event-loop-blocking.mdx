FastAPI를 처음 공부할 때 `def`와 `async def`의 차이를 문법 차이로 이해하면 곧바로 막힌다.
실제로 중요한 것은 문법이 아니라 **실행 모델**이다.

이 글은 FastAPI의 동기/비동기를 다음 질문으로 정리한다.

- 동기와 비동기의 차이는 정확히 무엇인가
- CPU, 프로세스, 스레드 관점에서 실행 흐름은 어떻게 다른가
- 코루틴과 이벤트 루프는 왜 "멈췄다가 다시 실행"될 수 있는가
- FastAPI에서 `def`와 `async def`는 실제로 어디서 실행되는가
- 어떤 코드가 blocking을 만들고, 언제 async가 오히려 독이 되는가
- 주니어 개발자가 실무에서 어떤 기준으로 sync/async를 선택해야 하는가

## 이 글의 전제

이 글은 다음 조합을 기준으로 설명한다.

- `CPython`
- `FastAPI`
- `asyncio`
- `uvicorn` 기반 ASGI 서버

구현 디테일은 런타임과 서버 설정에 따라 달라질 수 있지만, 실무 판단에 필요한 핵심 개념은 이 틀 안에서 거의 그대로 통한다.

## 1. 먼저 용어부터 분리하자

동기/비동기와 blocking/non-blocking은 자주 같이 언급되지만 같은 개념은 아니다.

- `동기(sync)`: 호출 흐름이 작업 완료와 강하게 결합된 모델
- `비동기(async)`: 작업 완료를 기다리는 동안 현재 흐름을 중단하고 다른 작업을 진행할 수 있는 모델
- `blocking`: 현재 실행 주체를 붙잡아 다음 작업을 못 하게 만드는 현상
- `non-blocking`: 현재 실행 주체를 붙잡지 않는 방식

실무에서는 보통 아래처럼 이해하면 충분하다.

- 동기 코드는 보통 현재 스레드가 기다린다.
- 비동기 코드는 보통 현재 코루틴이 기다리고, 스레드는 다른 코루틴 실행에 재사용된다.

## 2. CPU 관점에서 핵심은 "계산"보다 "대기 처리"다

서버 코드의 상당수 시간은 CPU 계산보다 대기에 쓰인다.

- DB 응답 대기
- 외부 API 응답 대기
- 파일 I/O 대기
- 네트워크 소켓 대기

이때 CPU는 실제 계산을 거의 하지 않는다.
병목은 "CPU가 느리다"가 아니라, **기다리는 동안 실행 흐름을 어떤 단위가 점유하느냐**에서 발생한다.

동기 모델에서는 대기를 스레드가 부담한다.
비동기 모델에서는 대기를 코루틴이 부담한다.

![sync vs async flow](/static/images/blog/fastapi-sync-async/sync-vs-async-flow.svg)

위 그림의 핵심은 단순하다.

- 동기 I/O는 현재 스레드를 묶는다.
- 비동기 I/O는 현재 코루틴만 멈추고 이벤트 루프가 다른 코루틴을 돌릴 수 있게 한다.

## 3. 프로세스, 스레드, 코루틴은 계층이 다르다

이 세 개를 섞어서 이해하면 async는 끝까지 헷갈린다.

- `프로세스`: 운영체제가 관리하는 독립 실행 단위
- `스레드`: 프로세스 안에서 실제로 CPU 스케줄링을 받는 단위
- `코루틴`: 언어 런타임과 이벤트 루프가 관리하는 사용자 공간 실행 단위

중요한 사실은 이거다.

**OS는 코루틴을 직접 스케줄링하지 않는다.**

OS가 CPU에 올리는 것은 프로세스/스레드다.
코루틴은 대개 **하나의 이벤트 루프 스레드 안에서 협력적으로 번갈아 실행**된다.

즉, 코루틴이 1000개 있어도 한 순간에 Python 바이트코드를 실행하는 것은 보통 이벤트 루프 스레드 위의 코루틴 하나다.

## 4. 코루틴은 왜 멈췄다가 다시 실행될 수 있을까

`async def`를 호출하면 함수가 즉시 끝까지 실행되지 않는다.
대신 **코루틴 객체**가 만들어진다.

이 코루틴 객체는 대략 다음 상태를 들고 있다.

- 어떤 코드를 실행 중인지
- 현재 어디까지 실행했는지
- 지역 변수 값이 무엇인지
- 다음에 어디서 재개해야 하는지
- 지금 무엇을 기다리고 있는지

예를 들어:

```python
async def foo():
    x = 10
    y = await fetch_value()
    return x + y
```

`await fetch_value()`에서 멈추는 순간에도 아래 상태는 사라지지 않는다.

- `x = 10`
- 다음 실행 위치
- `fetch_value()`의 완료 대기 상태

그래서 코루틴은 처음부터 다시 시작하는 것이 아니라 **중단된 위치 다음부터 재개**된다.

## 5. 이벤트 루프는 병렬 작업장이 아니라 스케줄러다

이벤트 루프를 "여러 칸이 있는 작업 공간"처럼 보면 오해가 생긴다.
보통 이벤트 루프 스레드는 한 번에 하나의 코루틴만 실제로 실행한다.
대신 아주 많은 대기 작업을 관리한다.

이벤트 루프는 대략 다음 일을 한다.

- 실행 가능한 Task를 고른다
- I/O 완료 이벤트를 감시한다
- 완료된 Task를 ready queue에 올린다
- 각 코루틴이 `await`를 만날 때까지 실행시킨다

여기서 중요한 포인트가 하나 있다.

**I/O 응답이 도착했다고 해서 해당 코루틴이 즉시 CPU를 잡는 것은 아니다.**

정확히는 "이제 다시 실행 가능한 상태가 됨"에 가깝다.
현재 돌고 있는 코루틴이 제어권을 넘기면 이벤트 루프가 그 Task를 다시 스케줄링한다.

![event loop resume timing](/static/images/blog/fastapi-sync-async/event-loop-resume.svg)

즉, async는 선점형이 아니라 보통 **협력형(cooperative)** 이다.
`await`를 만나야 다른 코루틴이 자연스럽게 끼어들 수 있다.

## 6. FastAPI에서 `def`와 `async def`는 실행 경로가 다르다

이제 FastAPI로 연결해보자.

### `async def` 엔드포인트

```python
@app.get('/users')
async def get_users():
    users = await async_repo.fetch_users()
    return users
```

이 코드는 이벤트 루프 위에서 코루틴으로 실행된다.
`await`를 만나면 현재 코루틴은 멈추고, 이벤트 루프는 같은 스레드에서 다른 코루틴을 실행할 수 있다.

### `def` 엔드포인트

```python
@app.get('/users')
def get_users():
    users = sync_repo.fetch_users()
    return users
```

일반 `def` 엔드포인트는 보통 스레드풀에서 실행된다.
즉, 이벤트 루프가 직접 동기 함수를 돌리는 것이 아니라 워커 스레드에 맡긴다.

이 차이를 이해해야 한다.

- `async def`: 이벤트 루프 + 코루틴 모델
- `def`: 스레드풀 + 동기 함수 모델

따라서 sync와 async는 단순 문법 선택이 아니라 **"어떤 실행 자원으로 대기 비용을 지불할 것인가"** 의 선택이다.

여기서 한 가지를 더 분명히 해야 한다.

**FastAPI의 `async def` 요청은 보통 "요청당 스레드 하나"로 처리되지 않는다.**
요청은 대개 이벤트 루프가 관리하는 Task로 올라가고, 같은 이벤트 루프 스레드 위에서 다른 요청 Task들과 번갈아 실행된다.

## 7. 코루틴, Task, thread pool은 무엇이 다른가

이 지점에서 개념을 하나 더 구분해야 한다.
실무에서는 `코루틴`, `Task`, `thread`를 거의 같은 말처럼 섞어 쓰지만, 실제로는 역할이 다르다.

### 코루틴 객체

`async def` 함수를 호출하면 먼저 **코루틴 객체**가 만들어진다.
이 객체는 "중단 가능한 함수 실행 상태"를 들고 있다.

```python
async def fetch_user():
    ...


coro = fetch_user()
```

위 코드에서 `coro`는 아직 스레드가 아니고, 실행 가능한 상태를 들고 있는 객체다.

### Task

Task는 이벤트 루프가 코루틴을 **스케줄링 가능한 작업 단위**로 감싼 것이다.
즉, 코루틴이 "실행 상태"라면 Task는 "이 코루틴을 언제 돌리고, 완료되면 어떤 결과를 낼지 관리하는 래퍼"에 가깝다.

그래서 비동기 서버에서 "요청 하나가 처리된다"는 말은 보통 더 정확히는 다음에 가깝다.

- 요청이 들어온다
- 해당 요청 처리 코루틴이 만들어진다
- 이벤트 루프가 그 코루틴을 Task로 관리하며 실행한다

### thread pool worker

반면 thread pool worker는 운영체제가 스케줄링하는 **실제 스레드**다.
FastAPI에서 `def` 엔드포인트나 기타 blocking 함수를 처리할 때는, 이벤트 루프가 직접 돌리지 않고 thread pool의 워커 스레드에 넘긴다.

즉:

- `async def` 경로: 코루틴 + Task + 이벤트 루프
- `def` 경로: 동기 함수 + thread pool worker

### 왜 이 차이가 중요한가

이 차이를 이해해야 아래 질문에 답할 수 있다.

- 왜 `async def` 요청은 많은 수를 적은 스레드로 처리할 수 있는가
- 왜 `def` 요청은 thread pool 크기에 영향을 받는가
- 왜 `async def` 안에 blocking 코드를 넣으면 이벤트 루프 전체가 느려질 수 있는가

### 비교표

| 항목          | 코루틴 객체                | Task                                 | thread pool worker                  |
| ------------- | -------------------------- | ------------------------------------ | ----------------------------------- |
| 정체          | 중단 가능한 함수 실행 상태 | 이벤트 루프가 관리하는 스케줄링 단위 | 운영체제가 스케줄링하는 실제 스레드 |
| 누가 관리하나 | Python 런타임              | 이벤트 루프                          | 운영체제 + 스레드풀                 |
| 주 사용 경로  | `async def` 호출 직후      | `async def` 요청 처리                | `def` 엔드포인트, blocking 함수     |
| 대기 처리     | 상태를 보존한 채 중단 가능 | `await`에서 다른 Task로 전환         | 스레드가 직접 기다림                |
| 비용          | 가벼움                     | 가벼움                               | 상대적으로 무거움                   |

실무적으로는 이렇게 이해하면 충분하다.

- 코루틴은 "중단 가능한 함수"
- Task는 "이벤트 루프가 돌리는 비동기 작업"
- thread pool worker는 "동기 코드를 대신 실행하는 실제 스레드"

따라서 FastAPI에서 `async def` 요청을 처리할 때는 보통 "스레드 하나가 요청 하나를 전담한다"가 아니라, **Task 여러 개가 하나의 이벤트 루프 스레드를 나눠 쓴다**고 이해하는 편이 정확하다.

## 8. blocking은 어디서 생기나

FastAPI에서 가장 흔한 실수는 `async def` 안에 blocking 코드를 넣는 것이다.

대표적인 예시는 다음과 같다.

```python
async def bad():
    time.sleep(5)
```

```python
async def bad():
    response = requests.get('https://example.com')
    return response.json()
```

```python
async def bad():
    rows = sync_db.execute('select * from users')
    return rows
```

위 코드는 모두 `async def` 안에 있지만 실제로는 동기 호출이다.
이 경우 현재 코루틴만 멈추는 것이 아니라 **이벤트 루프 스레드 자체가 묶일 수 있다.**

그 결과:

- 다른 코루틴의 재개가 지연된다
- 전체 처리량이 떨어진다
- tail latency가 급격히 나빠진다

즉, async의 핵심은 `async def` 선언이 아니라 **호출 체인 전체가 비동기 친화적인가** 에 있다.

## 9. "외부 API 호출" 자체가 문제가 아니라 "어떻게 기다리느냐"가 문제다

이 부분을 많이 오해한다.

외부 API 호출이 느린 것은 자연스럽다.
진짜 문제는 그 느린 시간을 누가 부담하느냐다.

### 나쁜 예: async 함수 안에서 동기 HTTP 호출

```python
import requests


@app.get('/bad')
async def bad():
    response = requests.get('https://api.example.com/data')
    return response.json()
```

이 경우 5초가 걸리면 이벤트 루프 스레드가 5초 동안 묶일 수 있다.

### 좋은 예: 비동기 HTTP 클라이언트 사용

```python
import httpx

client = httpx.AsyncClient()


@app.get('/good')
async def good():
    response = await client.get('https://api.example.com/data')
    return response.json()
```

이 경우 5초 동안 기다리는 것은 코루틴이다.
이벤트 루프는 그 동안 다른 요청을 처리할 수 있다.

## 10. async가 항상 더 빠른 것은 아니다

비동기의 장점은 "더 빠른 계산"이 아니라 **더 효율적인 대기 처리**다.

그래서 async는 주로 `I/O-bound` workload에서 강하다.

### async가 잘 맞는 경우

- 외부 API 호출이 많다
- DB 대기 시간이 길다
- WebSocket, SSE, streaming이 있다
- 동시 연결 수가 많다
- 하나의 요청이 여러 I/O를 조합한다

### async의 효과가 제한적인 경우

- 이미지 처리
- 압축
- 대규모 파싱
- 수치 계산
- 머신러닝 전처리

이런 작업은 `CPU-bound`다.
여기서는 이벤트 루프가 아니라 CPU 점유가 병목이므로, async만으로는 해결되지 않는다.

## 11. CPU-bound 작업은 왜 async로 해결되지 않을까

예를 들어 아래 코드는 `async def`지만 본질적으로 CPU 작업이다.

```python
async def heavy():
    total = 0
    for i in range(10**8):
        total += i
    return total
```

이 함수에는 의미 있는 `await`가 없다.
즉, 이벤트 루프가 다른 코루틴으로 전환할 지점이 없다.
결국 현재 스레드를 오래 점유하게 된다.

이런 경우 더 적절한 선택지는 다음과 같다.

- 별도 워커 프로세스
- 작업 큐
- 배치 처리
- 멀티프로세싱
- 외부 연산 서비스

## 12. sync vs async를 어떻게 선택할까

아래 기준으로 판단하면 된다.

| 상황                                         | 추천 방식                 | 근거                                              |
| -------------------------------------------- | ------------------------- | ------------------------------------------------- |
| 단순 CRUD, 낮은 트래픽, 동기 라이브러리 중심 | `def` 또는 동기 중심 구조 | 구조가 단순하고 운영 난이도가 낮다                |
| 외부 API 집약형 서비스                       | `async def`               | 네트워크 대기 시간을 코루틴 단위로 처리할 수 있다 |
| 채팅, WebSocket, 스트리밍                    | `async def`               | 연결 수가 많고 대기 시간이 길다                   |
| CPU 연산 중심 작업                           | 별도 워커/프로세스        | 병목이 I/O가 아니라 CPU 점유다                    |
| async와 sync 라이브러리가 혼재               | 경계 명확화               | async 경로와 blocking 경로를 분리해야 한다        |

## 13. 주니어 개발자가 특히 알아야 할 실무 포인트

### 1. `async def`만 붙이면 비동기가 되는 것이 아니다

이건 가장 흔한 오해다.
내부에서 `requests`, `time.sleep`, 동기 DB 드라이버를 호출하면 그 순간 blocking 코드가 된다.

### 2. "응답이 도착했다"와 "즉시 실행된다"는 다르다

응답 도착은 해당 Task가 ready 상태가 되었다는 뜻이다.
이벤트 루프가 현재 실행 중인 코루틴에서 제어권을 받아야 다시 실행된다.

### 3. CPU 작업이 긴 코루틴도 이벤트 루프를 망친다

blocking은 동기 I/O만 만드는 것이 아니다.
`await` 없이 오래 도는 계산 루프도 이벤트 루프를 점유한다.

### 4. async 경로에서는 라이브러리 선택이 설계의 일부다

예를 들면 다음 같은 차이가 생긴다.

- `requests` vs `httpx.AsyncClient`
- 동기 ORM 세션 vs 비동기 DB 세션
- `time.sleep` vs `asyncio.sleep`

즉, async는 함수 선언보다 **생태계 선택**의 문제다.

### 5. async는 처리량 최적화 도구이지 만능 성능 도구가 아니다

평균 응답시간만 보고 async를 도입하면 판단이 틀릴 수 있다.
처리량, 동시성, tail latency, downstream 병목까지 함께 봐야 한다.

### 6. 타임아웃, 취소, 재시도 정책이 더 중요해진다

비동기 서버는 동시에 많은 요청을 쉽게 날릴 수 있다.
그래서 타임아웃, circuit breaker, retry 정책이 없으면 오히려 장애 전파가 빨라질 수 있다.

### 7. DB connection pool과 외부 API rate limit을 반드시 고려해야 한다

이벤트 루프가 많은 Task를 관리할 수 있다고 해서 DB 연결과 외부 API 호출도 무한정 처리할 수 있는 것은 아니다.
병목은 종종 애플리케이션이 아니라 downstream 시스템에 있다.

## 14. 실무 체크리스트

FastAPI 코드에서 아래 질문에 하나라도 "아니오"가 나오면 async 경로를 다시 점검해야 한다.

- 이 `async def` 내부 호출은 모두 awaitable인가
- 동기 HTTP 클라이언트를 직접 호출하지 않는가
- 동기 DB 드라이버를 직접 호출하지 않는가
- 긴 CPU 연산을 요청 처리 경로에서 직접 수행하지 않는가
- 타임아웃과 예외 처리가 있는가
- connection pool 크기와 rate limit을 알고 있는가

## 15. 결론

FastAPI에서 동기와 비동기의 차이는 `def`와 `async def`의 문법 차이가 아니다.
핵심은 **대기 시간을 어떤 실행 단위가 부담하느냐**에 있다.

- 동기 모델에서는 스레드가 기다린다.
- 비동기 모델에서는 코루틴이 기다리고, 이벤트 루프는 다른 코루틴을 실행한다.
- `async def` 안의 blocking 코드는 이벤트 루프 전체를 늦출 수 있다.
- CPU-bound 문제는 async가 아니라 다른 실행 모델로 풀어야 한다.

결국 좋은 기준은 단순하다.

**I/O가 많으면 async를 검토하고, CPU가 많으면 실행 모델을 분리하며, 단순한 서비스라면 sync도 충분히 좋은 선택이다.**

실무에서 중요한 것은 "무조건 비동기"가 아니라, **내 서비스의 병목이 어디에 있는지 정확히 아는 것**이다.
