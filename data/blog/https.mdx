## HTTPS

<TOCInline toc={props.toc} exclude="HTTPS" />

HTTPS는 HTTP 위에 TLS를 적용해 통신을 보호하는 프로토콜이다. [HTTP 이해하기](/blog/http)

### SSL과 TLS

- SSL(Secure Sockets Layer): TLS 이전 세대 프로토콜 (현재는 사용 금지 수준)
- TLS(Transport Layer Security): SSL의 후속 표준, 현재 HTTPS의 실제 보안 계층
- 실무에서 "SSL 인증서"라고 부르는 경우가 많지만, 실제로는 대부분 TLS 인증서이다.

#### TLS 핵심 기능

1. 인증 (Authentication): 접속한 서버가 진짜 서버인지 인증서로 검증
2. 기밀성 (Confidentiality): 데이터 암호화로 도청 방지
3. 무결성 (Integrity): 전송 중 변경 여부를 검증

#### TLS 1.3 작동 방식

##### Handshake 과정

1. Client Hello (클라이언트 -> 서버): 브라우저가 서버에 지원 정보를 전달
    - 지원 가능한 TLS 버전
    - 지원하는 Cipher Suites
    - 클라이언트 난수(Random), Key Share 등

2. Server Hello (서버 -> 클라이언트): 서버가 사용할 알고리즘/파라미터를 선택

3. Certificate/CertificateVerify/Finished (서버 -> 클라이언트)
    - 서버 인증서 체인 전달
    - 서버가 개인키를 보유했음을 서명으로 증명

4. 인증서 검증 (클라이언트)
    - CA 체인, 도메인 일치, 유효기간 등을 검증
    - "CA 공개키로 인증서를 복호화"가 아니라, 서명과 체인을 검증하는 개념

5. Key Agreement 및 Client Finished
    - 양측은 (EC)DHE 기반으로 공유 비밀을 계산해 세션 키를 생성
    - 클라이언트 Finished까지 확인되면 핸드셰이크 완료

##### 데이터 전송

1. 암호화: 세션 대칭키로 데이터 암호화
2. 무결성 검증: TLS 1.3에서는 AEAD(AES-GCM, ChaCha20-Poly1305) 인증 태그로 검증
3. 검증 실패 시 폐기: 인증 태그가 맞지 않으면 데이터는 즉시 폐기

#### 대칭키와 비대칭키 역할

##### 대칭키

대칭키는 암호화/복호화에 같은 키를 사용한다.

- 장점: 속도가 빠름 (실제 데이터 전송에 적합)
- 단점: 키를 안전하게 공유해야 함
- 대표 알고리즘: AES-GCM, ChaCha20-Poly1305

##### 비대칭키

비대칭키는 공개키/개인키 쌍을 사용한다.

- 공개키(Public Key): 배포 가능한 키
- 개인키(Private Key): 서버만 보관하는 키
- 장점: 인증과 키 교환에 유리
- 단점: 대칭키보다 느림
- 대표 알고리즘: RSA/ECDSA(서명), ECDHE(키 합의)
- 참고: Diffie-Hellman 계열은 주로 "암호화 알고리즘"이 아니라 "키 교환 알고리즘"으로 분류한다.

## 꼬리물기 질문

1. HTTPS가 무엇인가요? -> HTTP와의 핵심 차이는? -> 기본 포트 번호는?
    - 답변: HTTPS는 HTTP에 TLS를 적용한 프로토콜이다. HTTP는 평문 통신이라 도청/변조/위장 위험이 있지만, HTTPS는 암호화와 인증서 검증으로 이를 줄인다. 기본 포트는 HTTP 80, HTTPS 443이다.

2. SSL과 TLS는 어떻게 다른가요? -> 왜 요즘은 SSL보다 TLS라고 하나요? -> 실무에서 "SSL 인증서"라고 부르는 이유는?
    - 답변: SSL은 구버전이고 TLS가 후속 표준이다. SSL 2.0/3.0은 취약점 때문에 현재는 사용하지 않는다. 실무에서 "SSL 인증서"라는 표현은 관성적으로 남아 있는 용어이고, 실제로는 TLS 인증서를 의미한다.

3. HTTPS에서 인증서가 왜 필요한가요? -> 브라우저는 인증서를 무엇으로 검증하나요? -> 인증서 만료 시 어떤 일이 생기나요?
    - 답변: 인증서는 서버 신원을 증명하고 공개키를 안전하게 전달하기 위해 필요하다. 브라우저는 신뢰 루트 CA 체인, 도메인 일치(SAN/CN), 유효기간을 검증한다. 만료되면 브라우저 경고/차단이 발생하고 사용자 이탈이 늘어난다.

4. 대칭키와 비대칭키 차이를 설명해보세요. -> HTTPS에서는 각각 어디에 쓰이나요? -> 왜 데이터 전송은 대칭키를 주로 쓰나요?
    - 답변: 대칭키는 같은 키로 암복호화하고, 비대칭키는 공개키/개인키 쌍을 쓴다. HTTPS에서는 비대칭키를 인증/키 합의에, 대칭키를 실제 데이터 전송에 사용한다. 대칭키가 속도가 훨씬 빨라 대용량 전송에 유리하기 때문이다.

5. TLS 핸드셰이크를 간단히 설명해보세요. -> ClientHello/ServerHello에서 무엇을 주고받나요? -> 핸드셰이크가 끝난 뒤 무엇이 달라지나요?
    - 답변: 클라이언트가 ClientHello로 버전/암호군/난수/키 공유 정보를 보내고, 서버가 ServerHello로 선택한 파라미터를 응답한다. 이어 인증서 검증과 키 합의를 거쳐 세션 키를 만든다. 핸드셰이크 이후에는 애플리케이션 데이터가 세션 키로 암호화되어 전송된다.

6. CA(인증기관) 신뢰 체인이 뭔가요? -> 중간 인증서가 왜 필요한가요? -> Self-signed 인증서는 왜 경고가 뜨나요?
    - 답변: 신뢰 체인은 서버 인증서 -> 중간 CA -> 루트 CA로 이어지는 검증 경로다. 중간 인증서는 루트 키 노출 위험을 줄이고 운영/발급을 분리하기 위해 쓴다. Self-signed는 브라우저 기본 신뢰 저장소에 연결된 체인이 없어 경고가 뜬다.

7. HTTPS면 완전히 안전한가요? -> 피싱 사이트도 HTTPS를 쓸 수 있나요? -> 사용자가 추가로 봐야 할 보안 신호는?
    - 답변: HTTPS는 전송 구간 보호 기술이지 사이트의 선의까지 보장하지 않는다. 피싱 사이트도 정상 인증서를 발급받아 HTTPS를 사용할 수 있다. 사용자는 도메인 철자, 발신자/링크 출처, 로그인 페이지 URL, 이상한 리다이렉트를 함께 확인해야 한다.

8. HTTP -> HTTPS 리다이렉트는 왜 설정하나요? -> 301과 302 중 보통 무엇을 쓰나요? -> 안 걸면 어떤 문제가 생기나요?
    - 답변: 사용자를 암호화 채널로 강제하고 URL 정규화(SEO)도 맞추기 위해 설정한다. 영구 이전이라면 보통 301을 사용한다. 설정이 없으면 일부 요청이 HTTP로 노출되어 세션 탈취/중간자 공격 위험이 커지고 중복 콘텐츠 이슈도 생긴다.

9. HSTS는 무엇이고 왜 쓰나요? -> HTTPS 강제와 리다이렉트만 있을 때와 차이는? -> 운영에서 주의할 점은?
    - 답변: HSTS는 브라우저가 일정 기간 해당 도메인을 HTTPS로만 접속하게 하는 정책이다. 단순 리다이렉트와 달리 최초 이후 HTTP 시도를 차단해 다운그레이드 공격(SSL Strip) 위험을 줄인다. 운영에서는 인증서 자동 갱신, 서브도메인 준비 여부, preload 적용 전 검증이 중요하다.

10. 배포 후 HTTPS 관련 장애가 나면 어디부터 보나요? -> 인증서/체인/도메인 불일치 중 무엇을 먼저 확인하나요? -> Mixed Content 오류는 어떻게 해결하나요?
    - 답변: 먼저 인증서 유효기간 만료 여부와 도메인 일치 여부를 확인하고, 그다음 인증서 체인 누락을 점검한다. 이후 리다이렉트/HSTS/프록시 설정을 본다. Mixed Content는 HTTP 리소스를 전부 HTTPS로 바꾸거나 동일 출처로 제공해 해결한다.
